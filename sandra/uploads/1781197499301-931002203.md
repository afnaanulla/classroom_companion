**Project**: Classroom Companion Telegram Bot (AI-Agentic System)  
**Timeline**: 2вҖ“3 days intensive development  
**Target Deadline**: Production-ready deployment with live demo capability  

---

## Executive Summary

This plan breaks the SIM assignment into **6 executable phases** spanning **3 days**, prioritizing:
1. **End-to-end correctness** (happy path working live)
2. **AI/LLM orchestration** (agents handling NLP intelligently)
3. **System resilience** (no dead code, no fallbacks, meaningful error handling)
4. **Production readiness** (deployable, persistent, scalable)
5. **Demo-ready state** (extensible for interview pair-programming)

## Phase Breakdown (6 Phases over 3 Days)

### PHASE 1: Foundation & Architecture Setup
**Duration**: Day 1, Morning (~4 hours)  
**Goal**: Project skeleton, database schema, core abstractions in place  

#### Deliverables
1. **Project Structure**
   - Backend folder with `src/` containing:
     - `agents/` (one file per agent: intent, teacher, student, reminder, summarizer)
     - `services/` (business logic: assignment, submission, feedback, reminder managers)
     - `models/` (database models/schemas)
     - `telegram/` (bot handlers, command routers)
     - `utils/` (logging, config, LLM provider abstraction)
     - `types/` (TypeScript interfaces for messages, assignments, users)
   - Frontend folder with `src/` containing:
     - `pages/` (teacher-dashboard, student-dashboard)
     - `components/` (reusable UI blocks)
     - `hooks/` (API integration, state)
     - `types/` (API response shapes)
   - Database migrations folder
   - `.env.example` (no secrets)

2. **Database Schema**
   - `users` table: id, telegramHandle, role (teacher/student), registeredAt, verificationCode
   - `teachers` table: userId (FK), name, email, metadata
   - `students` table: userId (FK), name, teacherId (FK), linkedAt
   - `assignments` table: id, teacherId (FK), studentId (FK), title, description, dueDate, createdAt, status
   - `submissions` table: id, assignmentId (FK), studentId (FK), textContent, filePath, submittedAt
   - `feedback` table: id, submissionId (FK), teacherFeedback, deliveredAt, readAt
   - `reminders` table: id, studentId (FK), assignmentId (FK), scheduledFor, sentAt, reminderType (daily/escalating)
   - `agent_logs` table: id, agentType, input, output, executedAt (for debugging)

3. **Core Abstractions**
   - LLM Provider Interface (abstract layer for Claude/GPT/Mistral swapping)
   - Message Types (IncomingMessage, AgentResponse, BotCommand, BotMessage)
   - Agent Interface (base class/interface for all agents)
   - Error Handling (meaningful error types: ValidationError, ConflictError, NotFoundError, ExternalServiceError)
   - Logger (structured logging with context)
   - Config Manager (environment-based, with validation)

4. **Setup & Dependencies**
   - Backend: Express, telegraf, @anthropic-ai/sdk, typeorm/prisma, bull/bullmq, dotenv
   - Frontend: React, axios, react-router, TypeScript
   - Dev tools: ESLint, Prettier, ts-node, nodemon
   - Testing framework installed (Jest or Vitest) вҖ” not yet used

5. **Git Setup**
   - Initial commit with folder structure
   - `.gitignore` with `.env`, `node_modules/`, database files
   - Remote GitHub repo created
   - No code, just scaffolding and empty files with comments

### PHASE 2: Telegram Bot & User Onboarding
**Duration**: Day 1, Late Morning вҶ’ Afternoon (~5 hours)  
**Goal**: Bot receives messages, commands work, teacher & student can register and link  

#### Deliverables
1. **Telegram Bot Setup**
   - Bot token stored in `.env`
   - Webhook or polling listener active
   - Message handler that logs all incoming messages
   - Error recovery (connection failures logged, not crashed)

2. **User Registration Commands**
   - `/start` command вҶ’ detects new user, asks for role (teacher/student)
   - `/register_teacher` вҶ’ collects name, email (if desired), stores in database
   - `/register_student` вҶ’ collects name, asks for teacher link code
   - `/link_teacher <code>` вҶ’ validates code, links student to teacher, stores relationship

3. **Teacher Onboarding Flow**
   - Teacher receives a unique 6-digit linking code on registration
   - Code displayed in bot message and stored in database
   - Code expires after 24 hours or 5 uses (whichever first)
   - Teacher can generate new code via `/new_code`

4. **Student Linking Flow**
   - Student enters code via `/link_teacher <code>`
   - Bot validates code, checks expiry, checks if code belongs to a teacher
   - On success: relationship stored, student sees confirmation with teacher name
   - On failure: clear error message (expired, invalid, used too many times)

5. **Message Routing**
   - All non-command messages routed to Intent Router Agent
   - Router classifies: assignment_instruction, status_update, completion_notice, query, other
   - Logs for debugging

6. **Data Validation**
   - All user inputs validated (length, format, special characters)
   - Database constraints (unique Telegram handles, valid relationships)
   - No stale data (expired codes cleaned up)

### PHASE 3: Agent Orchestration Layer
**Duration**: Day 1, Late Afternoon вҶ’ Evening (~5 hours)  
**Goal**: All 5 agents defined, callable, returning structured responses  

#### Agents & Responsibilities

**3A. Intent Router Agent**
- **Input**: Raw Telegram message (from student or teacher)
- **Output**: Classification object with:
  - `intent` (enum: ASSIGNMENT_INSTRUCTION, STATUS_UPDATE, COMPLETION_NOTICE, QUERY, FEEDBACK_REQUEST, OTHER)
  - `confidence` (0-1)
  - `extracted_entities` (names, dates, durations mentioned)
  - `fallback_message` (if intent unclear, what should bot say)
- **Prompt Strategy**: Few-shot examples of each intent category, no hallucination
- **Handling**: If confidence < 0.6, ask user to clarify; never guess

**3B. Teacher Agent**
- **Inputs**:
  - Incoming message from teacher (verified via database check)
  - Intent classification (from router)
  - Teacher context (students list, active assignments)
- **Outputs**:
  - `assignment_object` (if ASSIGNMENT_INSTRUCTION): title, description, due date, target student
  - `feedback_text` (if FEEDBACK_REQUEST): personalized feedback for student
  - `confirmation_message` (what to send back to teacher)
  - `errors` (validation issues)
- **Prompt Strategy**: Clarify ambiguities ("which student?", "parse 'in 3 days' вҶ’ specific date"), extract due dates robustly
- **Handling**: If any required field missing, ask teacher explicitly (don't assume)

**3C. Student Agent**
- **Inputs**:
  - Incoming message from student (verified via database check)
  - Intent classification (status update or completion)
  - Student context (active assignments, past submissions)
- **Outputs**:
  - `status_update` (if STATUS_UPDATE): which assignment, what progress, sentiment (stuck/on-track/ahead)
  - `submission_object` (if COMPLETION_NOTICE): text, file references, timestamp
  - `confirmation_message` (what to send back to student)
  - `errors` (validation)
- **Prompt Strategy**: Interpret natural language progress ("done 2 paragraphs" вҶ’ 40% done, "stuck on intro" вҶ’ need help signal)
- **Handling**: Confirm submission received; never lose student submissions

**3D. Reminder/Scheduler Agent**
- **Inputs**:
  - Assignment object (new or existing)
  - Current time
  - Historical reminder sends (to avoid spamming)
- **Outputs**:
  - `reminder_schedule` (list of [datetime, message_template, student_id, assignment_id])
  - `escalation_level` (1=daily, 2=daily+urgent flag, 3=critical)
  - `first_reminder_delay` (e.g., send first reminder in 1 day if due in 3 days)
- **Prompt Strategy**: No hardcoded intervals; let LLM decide thoughtful timing based on deadline distance
- **Handling**: Store schedule in database, job queue picks up and executes

**3E. Summarizer Agent**
- **Inputs**:
  - List of students (for teacher)
  - Their active assignments
  - Recent status updates, submissions, feedback sent
  - Time period (last 24h, week, etc.)
- **Outputs**:
  - `summary_per_student` (Riya: on-track with essay, waiting for feedback on math; Arjun: stuck on project, check in)
  - `alerts` (students overdue, no recent activity, submissions pending review)
- **Prompt Strategy**: Narrative, not just data dump; highlight exceptions
- **Handling**: Generate on-demand (when teacher asks `/status`) or periodically (daily digest)

#### Orchestration Pattern
- Single orchestrator function: `processMessage(telegramMessage, user) вҶ’ botResponse`
- Flow:
  1. Validate user exists and is active
  2. Call Intent Router with message
  3. Route to appropriate agent (Teacher/Student/etc.) based on intent
  4. Agent processes, returns structured output
  5. Orchestrator calls Business Logic layer to persist changes
  6. Return bot message to Telegram
  7. Log full flow (input вҶ’ agent вҶ’ output вҶ’ DB) for debugging

### PHASE 4: Business Logic & Persistence
**Duration**: Day 2, Morning вҶ’ Afternoon (~6 hours)  
**Goal**: All CRUD operations for assignments, submissions, feedback, reminders; data survives restarts  

#### Deliverables

**4A. Assignment Manager Service**
- `createAssignment(teacher_id, student_id, title, description, due_date)` вҶ’ stored, returns ID
- `getAssignmentsByTeacher(teacher_id)` вҶ’ all assignments for that teacher
- `getAssignmentsByStudent(student_id)` вҶ’ all assignments for that student
- `updateAssignmentStatus(assignment_id, new_status)` вҶ’ PENDING вҶ’ IN_PROGRESS вҶ’ SUBMITTED вҶ’ FEEDBACK_GIVEN
- `getOverdueAssignments()` вҶ’ for reminder engine
- Validation: due_date in future, student linked to teacher, no duplicate assignments

**4B. Submission Handler Service**
- `createSubmission(assignment_id, student_id, text_content, file_paths)` вҶ’ stores submission
- `getSubmissionsByAssignment(assignment_id)` вҶ’ ordered by timestamp
- `getSubmissionsByStudent(student_id)` вҶ’ all submissions from that student
- `markSubmissionAsReviewed(submission_id)` вҶ’ teacher has seen it
- Validation: only one submission per assignment (or allow revisions with versioning)
- File handling: store file paths securely, assume files uploaded separately (not in message)

**4C. Feedback Manager Service**
- `createFeedback(submission_id, teacher_id, feedback_text)` вҶ’ stores feedback
- `getFeedback(submission_id)` вҶ’ retrieve for student
- `markFeedbackAsRead(feedback_id)` вҶ’ student has read it
- `getFeedbackHistory(student_id)` вҶ’ all feedback student ever received
- Validation: feedback only from teacher of linked student

**4D. Reminder Engine Service**
- `scheduleReminder(assignment_id, student_id, schedule_list)` вҶ’ creates reminder records
- `getPendingReminders()` вҶ’ reminders due to send now
- `markReminderAsSent(reminder_id)` вҶ’ prevents duplicate sends
- `getReminders(assignment_id)` вҶ’ see all reminders for an assignment
- Escalation logic: if student hasn't responded, increase urgency in next reminder
- Validation: don't double-send same reminder

**4E. Status Aggregator Service**
- `getTeacherStatus(teacher_id, time_period?)` вҶ’ summary per student
- `getStudentStatus(student_id)` вҶ’ own assignments + feedback
- Uses Summarizer Agent to format
- Caching: short TTL (5 min) to avoid re-querying LLM constantly

**4F. Database Persistence**
- TypeORM or Prisma migrations applied on startup
- Indexes on: (teacher_id, student_id), (assignment_id), (due_date), (user_id)
- Transactions: when creating assignment вҶ’ also create first reminder, all atomic
- No partial failures (if submission saved but feedback fails, roll back)
- Connection pooling for efficiency

## Technology Stack Deep Dive

### Backend Architecture

**Framework: Node.js + Express.js + TypeScript**
- Express for HTTP server (bot webhook, API endpoints)
- TypeScript for type safety, catch errors at compile time
- Hot reload in dev (nodemon)

**Agent Orchestration: LangGraph (LangChain) OR Plain Claude SDK**
- **Option A (Recommended for simplicity)**: Plain Claude SDK
  - Direct API calls, no framework overhead
  - Full control over prompts
  - Easier to swap providers
  - Suitable for fresher-level understanding
  
- **Option B (For bonus points)**: LangGraph
  - Structured agent orchestration with state persistence
  - Built-in retry, branching logic
  - More "production-grade" appearance
  - Steeper learning curve

**Database: PostgreSQL (Production) / SQLite (Local Dev)**
- TypeORM or Prisma for ORM (type-safe queries)
- Migrations versioned in Git
- Connection pooling with max 10 connections
- Prepared statements to prevent SQL injection

**Job Queue: Bull (Node.js) or Celery (Python)**
- Separate worker process for reminders
- Persistent job storage in Redis
- Retry logic built-in
- Named jobs for clarity: "sendReminder", "generateStatus", etc.

### Frontend Architecture

**Framework: React 18 + TypeScript + Vite**
- Vite for fast HMR (hot module replacement)
- React hooks for state management (no Redux unless necessary)
- TypeScript strict mode for type safety

**Styling: Tailwind CSS**
- Utility-first, minimal custom CSS
- Pre-built color palette, no design decisions needed
- Responsive grid for dashboards

**HTTP Client: Axios + Custom Hooks**
- Axios for requests (better error handling than fetch)
- Custom React hooks: `useTeacher()`, `useStudent()`, `useAssignments()`
- Loading, error, data states handled in hooks

### Deployment & DevOps

**Containerization: Docker**
- Single Dockerfile for backend
- docker-compose.yml for local development
- .dockerignore to reduce image size

**CI/CD: GitHub Actions**
- Run tests on every push
- Lint code
- Build Docker image on main branch merge
- Deploy to Railway/Render automatically

**Environment Management: 12-Factor App**
- All config in environment variables
- `.env.example` committed, `.env` never committed
- No hardcoded secrets anywhere

## Key Design Decisions & Rationales

| Decision | Rationale |
|----------|-----------|
| **Plain Claude SDK (not LangGraph)** | Simpler, full control, easier to explain in interview, sufficient for fresher level |
| **PostgreSQL over SQLite** | Durability, concurrency support, scales for production |
| **Bull (Node.js) for jobs** | In-process, no external service, easy to debug |
| **React + Vite** | Fast HMR, modern tooling, type-safe with TS |
| **TypeScript strict mode** | Catch errors at compile time, self-documenting code |
| **Meaningful names** | `Intent` not `I`, `AssignmentManager` not `AM`, `sendReminder` not `sr` |
| **Separation of concerns** | Agents don't touch DB, services don't call Telegram, each layer independent |
| **Provider abstraction** | LLMProvider interface allows Claude вҶ’ GPT swap in 5 minutes (interview win) |
| **Early failing** | Validate input immediately, don't attempt processing bad data |

## Known Limitations & Future Improvements

### Limitations (Document in README)
1. **Auth**: Simple linking code, no passwords or 2FA (acceptable for demo)
2. **Concurrency**: Basic locking (row-level), not distributed locks for truly concurrent edits
3. **Storage**: Files uploaded to `/uploads` on server, not cloud (S3/GCS)
4. **Scaling**: Single server instance, no load balancing
5. **LLM**: No caching of repeated queries; each request hits API
6. **UI**: No real-time updates (polling only), not WebSocket
7. **Mobile**: Web UIs not optimized for mobile (ok per spec)

## Common Pitfalls to Avoid

1. вқҢ **Dead code**: Remove print statements, commented-out code, unused imports
2. вқҢ **Hardcoded values**: No bot tokens, API keys, DB URLs in code (use .env)
3. вқҢ **Repeated logic**: DRY principleвҖ”extract common functions to utils
4. вқҢ **Generic error handling**: Catch specific errors, return meaningful messages
5. вқҢ **Poor naming**: `data`, `temp`, `x`, `fn` вҶ’ instead use `assignmentObject`, `tempReminder`, `userId`, `processMessage`
6. вқҢ **No validation**: Check all user inputs; don't trust Telegram/UI
7. вқҢ **One big commit**: Commit often (end of each feature/phase)
8. вқҢ **No tests**: Even basic test structure shows production-mindedness
9. вқҢ **UI/UX polish over functionality**: Get it working first, then polish
10. вқҢ **Ignoring the rubric**: Every component maps to a rubric item; optimize for points
